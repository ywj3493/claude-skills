# Project Instructions

This project follows a structured documentation system. All work is driven by
and recorded in the `docs/` directory. Claude must treat `docs/` as the source
of truth and keep it up to date.

## Documentation Structure

```text
docs/
├── specifications/   # Architecture, infrastructure, requirements, user stories, use cases, sequence diagrams
├── issue/            # One file per work item: issue001.md, issue002.md, ...
├── reference/        # Reference materials — user-provided only, never generated by Claude
├── policy/           # Working rules: commit messages, naming conventions, general policy
└── dev/              # Korean translations of all docs (except reference/)
    ├── specifications/
    ├── issue/
    └── policy/
```

## Core Rules

1. **Start with an issue.** Every piece of work — feature, bug fix, refactor,
   investigation — begins by creating or referencing an issue document in
   `docs/issue/`. Do not start implementation without one.

2. **English first.** All docs in `docs/` (except `docs/dev/`) are written in
   English.

3. **Korean mirror.** Every doc created in `docs/specifications/`, `docs/issue/`,
   and `docs/policy/` gets a matching Korean translation in `docs/dev/` with a
   `.ko.md` suffix.
   Example: `docs/issue/issue003.md` → `docs/dev/issue/issue003.ko.md`

4. **Reference is read-only.** Never create or modify files in `docs/reference/`.
   That directory is managed exclusively by the user.

5. **Policy is law.** Before starting any task, load all @-referenced
   policy files. Policy files take precedence over general conventions.

6. **Keep docs current.** When implementation changes, update the relevant docs
   in the same commit or PR.

## Required Context

Load these files before starting any work:

- [@docs/policy/policy.md](docs/policy/policy.md) — General working rules
- [@docs/policy/commit-message-rule.md](docs/policy/commit-message-rule.md) — Commit message format
- [@docs/policy/naming-conventions.md](docs/policy/naming-conventions.md) — File, variable, and branch naming
- [@docs/policy/reference-convention.md](docs/policy/reference-convention.md) — Document linking convention

For active work, also load the relevant issue from `docs/issue/`.

## Issue Document Format

Each issue document must contain:

- **Title** and brief description
- **Status** — Open / In Progress / Done
- **Background** — why this work is needed
- **Acceptance Criteria** — what "done" looks like (checkboxes)
- **Tasks** — numbered checklist of steps
- **Notes** — decisions and discoveries made during implementation

## Language Notes

- Code, variable names, file paths, technical identifiers: English
- All documentation in `docs/` except `docs/dev/`: English
- All documentation in `docs/dev/`: Korean
- Code blocks inside Korean docs remain in English
